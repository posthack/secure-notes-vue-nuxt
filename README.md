# Secure Notes

Заметки со сквозным шифрованием. Всё шифруется у вас в браузере мастер-паролем, на сервер уходит только шифртекст. Прочитать заметки нельзя, даже имея полный доступ к базе.

Пет-проект. Мне было интересно не наклепать фич, а собрать честную клиентскую криптографию: аккуратную модель ключей и внятную threat model, без «шифрования at-rest» под видом E2E.

## Демо

[Живое демо](#) — скоро выложу. Аккаунт там не нужен: заходишь, жмёшь «демо-данные» и смотришь на паре заметок. Шифрование и расшифровка идут в браузере, ключи с устройства не уходят. Синхронизация в демо выключена — она работает при self-host (см. ниже).

## Как это шифруется

- Мастер-пароль → PBKDF2 (600k итераций, соль на пользователя) → мастер-ключ. Argon2id был бы честнее, но тянет WASM — оставил PBKDF2 как осознанный компромисс.
- Мастер-ключ не шифрует заметки напрямую. Он оборачивает (AES-KW) случайный ключ, свой на каждую заметку. Поэтому смена пароля переворачивает только обёртки ключей, а не перешифровывает все данные.
- Каждая заметка шифруется AES-256-GCM со своим IV, id заметки идёт в AEAD — подменить метаданные незаметно не выйдет, расшифровка сломается.
- Ключи держатся в памяти веб-воркера, не в `localStorage`. При выходе стираются, плюс авто-лок через 5 минут простоя.

Крипто-ядро написано по TDD, тесты покрывают round-trip, смену пароля и порчу шифртекста. Полный разбор, от чего защищаемся и от чего нет, — в [SECURITY.md](./SECURITY.md).

## Синхронизация

Аккаунт (Better Auth, почта и пароль) нужен только чтобы синкать заметки между устройствами. Пароль от аккаунта и мастер-пароль — разные вещи: сервер хранит зашифрованные blob'ы и мастер-пароль в глаза не видит. Разрешение конфликтов простое, last-write-wins по времени правки.

## Шаринг

Заметкой можно поделиться с другим пользователем, и сервер по-прежнему её не прочитает. У каждого своя пара ключей RSA-OAEP-2048: публичный лежит открыто, приватный — под мастер-ключом. Чтобы поделиться, ключ заметки заворачивается публичным ключом получателя; расшифровать обёртку может только он своим приватным. Сервер видит шифртекст заметки и шифртекст обёртки, ключа под ними нет.

Уходит снимок: правки после шаринга к получателю не едут, надо поделиться заново. Шар можно ограничить сроком и отозвать.

## Вложения

К заметке можно прикрепить файл. Шифруется он так же, как заметка: свой ключ на файл, обёрнутый мастер-ключом. Имя, тип и размер лежат в отдельном зашифрованном блоке, сервер их не видит. Содержимое режется на чанки по 256 КБ, каждый чанк — свой IV и своя привязка к файлу и позиции, так что переставить или подменить кусок незаметно нельзя.

Список файлов синкается вместе с метаданными, а сами чанки сервер отдаёт по запросу — когда файл реально открывают. Одного полностью честно скрыть не выходит: по числу и длине чанков виден примерный размер файла. Точное значение прячу, порядок величины — нет; для заметочника счёл это допустимым.

## Запуск

```bash
npm install
cp .env.example .env    # проставить BETTER_AUTH_SECRET
npm run db:migrate      # накатить схему в локальный sqlite
npm run dev             # http://localhost:3000
```

Остальное:

```bash
npm run build       # прод-сборка
npm run generate    # статическая витрина (демо-режим)
npm test            # тесты
npm run typecheck
npm run lint
```

## Что осталось и чего сознательно нет

Готовы крипто-ядро, локальные заметки, синхронизация, шифрованные вложения и шаринг. Есть авто-лок, генератор паролей и [SECURITY.md](./SECURITY.md) с threat model. Незакрытое: строгий CSP-заголовок и живой деплой.

От чего проект не защищает и это нормально: XSS на странице убивает любой E2E (пока нет CSP — это дыра, которую шифрование не закрывает), скомпрометированное устройство, слабый мастер-пароль. Синк пока теряет удаления, сделанные оффлайн, — локально не веду тумбстоуны. Подробности и остальные компромиссы — в SECURITY.md.
